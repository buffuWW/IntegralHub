# Техническое задание для Codex: Integral Hub

Создай полноценное веб-приложение **Integral Hub** — русскоязычную образовательную платформу с каталогом заданий по интегралам.

Не создавай только визуальный макет. Реализуй работающее приложение с базой данных, серверными обработчиками, защищённой административной панелью, импортом CSV/ZIP, хранением пользовательского прогресса и демонстрационными заданиями.

---

# 1. Назначение проекта

Integral Hub предназначен для самостоятельной подготовки по интегральному исчислению.

Пользователь выбирает задание, решает его самостоятельно на бумаге, после чего может:

* посмотреть краткий ответ;
* посмотреть подробное решение;
* отметить задание как решённое правильно;
* отметить задание как требующее повторения;
* открыть следующее или предыдущее задание;
* получить случайное задание;
* просмотреть историю;
* повторить задания, с которыми возникли трудности.

Сайт не должен автоматически вычислять интегралы.

Пользователь не вводит:

* собственный интеграл;
* собственное решение;
* числовой или символьный ответ.

Все задания, ответы и решения заранее находятся в базе данных и добавляются администратором.

# 2. Ограничения первой версии

В первой версии не нужны:

* регистрация пользователей;
* авторизация обычных пользователей;
* личный кабинет;
* синхронизация прогресса между устройствами;
* проверка введённых пользователем ответов;
* таймер;
* экзаменационный режим;
* готовые контрольные работы;
* тёмная тема;
* английская версия;
* автоматическое решение интегралов;
* разделение заданий по методам интегрирования.

Интерфейс должен быть только на русском языке.

Используй:

* TypeScript со строгой типизацией;
* React;
* PostgreSQL;
* Prisma ORM;

* KaTeX;

# 5. Роли

## 5.1 Обычный пользователь

Работает без регистрации.

Может:

* просматривать опубликованные задания;
* использовать фильтры;
* открывать ответы;
* открывать решения;
* отмечать результат решения;
* просматривать свою историю;
* повторять сложные задания;
* получать случайное задание.

Обычный пользователь не должен иметь доступа к административным маршрутам и скрытым заданиям.

## 5.2 Администратор

В системе один администратор.

Администратор может:

* авторизоваться;
* просматривать статистику;
* создавать задания вручную;
* редактировать задания;
* скрывать и публиковать задания;
* перемещать задания в архив;
* восстанавливать задания из архива;
* импортировать задания через CSV;
* импортировать изображения вместе с заданиями;
* создавать категории;
* редактировать категории;
* скрывать категории;
* просматривать результаты импорта.

# 6. Категории интегралов

При первоначальном наполнении создай категории:

1. Неопределённые интегралы.
2. Определённые интегралы.
3. Несобственные интегралы.
4. Двойные интегралы.
5. Тройные интегралы.
6. Криволинейные интегралы.
7. Поверхностные интегралы.
8. Интегралы с параметром.
9. Другие интегралы.

У каждой категории должны быть:

* название;
* уникальный slug;
* краткое описание;
* порядок отображения;
* состояние активности;
* дата создания;
* дата изменения.

Администратор должен иметь возможность добавлять новые категории без изменения исходного кода.

Категории, в которых нет опубликованных заданий, можно показывать неактивными с количеством заданий 0.

---

# 7. Уровни сложности

Используй три уровня:

* `EASY` — лёгкий;
* `MEDIUM` — средний;
* `HARD` — сложный.

В пользовательском интерфейсе:

* лёгкий уровень обозначается зелёной меткой;
* средний — оранжевой;
* сложный — красной.

Цвет не должен быть единственным способом определения сложности. Всегда выводи текстовое название.

---

# 8. Состояния задания

Используй следующие серверные состояния:

* `PUBLISHED` — опубликовано;
* `HIDDEN` — скрыто;
* `ARCHIVED` — находится в архиве.

Обычный пользователь видит только `PUBLISHED`.

Скрытое задание:

* остаётся в базе;
* доступно администратору;
* не показывается пользователям;
* может быть снова опубликовано.

Архивное задание:

* не показывается пользователям;
* отображается в разделе «Архив»;
* может быть восстановлено в течение 24 часов;
* после 24 часов удаляется окончательно вместе с относящимися к нему изображениями.

Срок отсчитывается с момента перемещения в архив, а не до конца календарного дня.

# 9. Структура публичного сайта

## 9.1 Общая шапка

Шапка должна содержать:

* логотип или текстовый знак Integral Hub;
* ссылку «Главная»;
* ссылку «Все задания»;
* ссылку «История»;
* ссылку «Повторение»;
* кнопку «Случайное задание».

На мобильных устройствах навигация должна открываться через компактное меню.

Ссылку на административную панель не нужно делать заметной в основной навигации. Администратор открывает `/admin`.

## 9.2 Главная страница `/`

Главная страница должна содержать:

### Первый экран

* название Integral Hub;
* подзаголовок о самостоятельной практике интегралов;
* кнопку «Начать решать»;
* кнопку «Случайное задание»;
* ненавязчивый фон с математическими символами.

Пример смысла текста:

«Решайте интегралы самостоятельно, проверяйте ответы и возвращайтесь к заданиям, которые требуют повторения».

### Блок категорий

Показывать карточки категорий:

* название;
* краткое описание;
* количество опубликованных заданий;
* кнопку перехода.

### Блок сложности

Кратко объяснить три уровня сложности.

### Статистика базы

Показать:

* общее число опубликованных заданий;
* количество категорий;
* количество лёгких, средних и сложных заданий.

### Блок «Как заниматься»

Три простых шага:

1. Выберите задание.
2. Решите его самостоятельно.
3. Проверьте ответ и отметьте результат.

## 9.3 Каталог `/tasks`

Страница содержит:

* заголовок;
* количество найденных заданий;
* строку поиска;
* фильтр категории;
* фильтр сложности;
* фильтр пользовательского статуса;
* переключатель «Показывать уже решённые»;
* сортировку;
* кнопку сброса фильтров;
* кнопку «Случайное по фильтрам».

Поддерживаемая сортировка:

* сначала новые;
* сначала старые;
* по номеру по возрастанию;
* по номеру по убыванию;
* сначала лёгкие;
* сначала сложные.

Фильтры должны сохраняться в query-параметрах URL.

Пример:

`/tasks?category=double-integrals&difficulty=medium&solved=false`

Поиск должен работать по:

* номеру задания;
* условию;
* формуле;
* названию категории.

Добавь пагинацию. По умолчанию показывай 12 заданий на странице.

## 9.4 Карточка в каталоге

Карточка должна показывать:

* «Задание №…»;
* категорию;
* сложность;
* краткое условие;
* основную формулу;
* пользовательский статус;
* кнопку «Открыть задание».

Полное решение и ответ непосредственно в каталоге не показывать.

## 9.5 Страница задания `/tasks/[number]`

Страница должна содержать:

* номер;
* категорию;
* сложность;
* условие;
* основную формулу;
* изображения и чертежи;
* источник, если он указан;
* кнопку «Показать ответ»;
* кнопку «Показать решение»;
* кнопки оценки результата;
* кнопку предыдущего задания;
* кнопку следующего задания;
* кнопку случайного задания.

Ответ и решение должны быть скрыты по умолчанию.

Кнопки работают независимо:

* «Показать ответ» открывает только ответ;
* «Показать решение» открывает подробное решение;
* после открытия кнопка меняется на «Скрыть ответ» или «Скрыть решение».

Открытие ответа не должно автоматически открывать решение.

Открытие решения должно сохраняться в локальной истории.

Формулы отображать через KaTeX.

Решение отображать как Markdown с поддержкой LaTeX.

Не разрешать выполнение HTML или JavaScript из Markdown.

## 9.6 Случайное задание

На главной странице кнопка выбирает случайное опубликованное задание из всей базы.

В каталоге кнопка «Случайное по фильтрам»:

* учитывает выбранную категорию;
* учитывает сложность;
* учитывает пользовательские статусы;
* по умолчанию исключает задания со статусом «Решено правильно».

Если подходящих нерешённых заданий нет, показать сообщение и предложить:

* включить решённые задания;
* сбросить фильтры;
* выбрать другую категорию.

Не возвращать скрытые и архивные задания.

## 9.7 История `/history`

История хранится в браузере.

Показывать:

* недавно открытые задания;
* дату и время последнего просмотра;
* открывался ли ответ;
* открывалось ли решение;
* текущий статус пользователя.

Добавить фильтры:

* все;
* решено правильно;
* нужно повторить;
* без оценки.

Добавить кнопку «Очистить историю».

Перед очисткой показать подтверждение.

Очистка истории не обязана очищать статусы. Сделай отдельные действия:

* «Очистить историю просмотров»;
* «Сбросить весь прогресс».

Для полного сброса обязательно запросить подтверждение.

## 9.8 Повторение `/review`

На странице находятся задания, отмеченные как «Нужно повторить».

Показывать:

* количество таких заданий;
* фильтры по категории и сложности;
* список карточек;
* кнопку «Начать повторение».

Режим повторения:

* показывает по одному заданию;
* выбирает задания в случайном порядке;
* не использует таймер;
* не является экзаменом;
* позволяет открыть ответ и решение;
* позволяет отметить «Решил правильно»;
* позволяет оставить «Нужно повторить»;
* показывает оставшееся количество заданий.

После отметки «Решил правильно» задание должно исчезать из списка повторения.

# 10. Локальный пользовательский прогресс

Используй localStorage.

Для каждого задания сохраняй:

* идентификатор или номер;
* дату первого просмотра;
* дату последнего просмотра;
* количество просмотров;
* открывался ли ответ;
* открывалось ли решение;
* пользовательский статус.

Пользовательские статусы:

* `UNSEEN` — ещё не открывалось;
* `VIEWED` — просмотрено, но не оценено;
* `SOLVED` — решено правильно;
* `REVIEW` — нужно повторить.

Создай версионированную структуру данных, например:

```json
{
  "version": 1,
  "tasks": {
    "125": {
      "status": "REVIEW",
      "firstViewedAt": "2026-06-17T12:00:00.000Z",
      "lastViewedAt": "2026-06-17T12:30:00.000Z",
      "viewCount": 2,
      "answerOpened": true,
      "solutionOpened": true
    }
  }
}
```

Доступ к localStorage должен быть вынесен в отдельный модуль.

Учти серверный рендеринг: не обращайся к `window` и localStorage на сервере.

История привязана только к текущему браузеру. Не создавай ложного впечатления, что она синхронизируется между устройствами.

# 11. Кнопки оценки результата

На странице задания размести две заметные кнопки:

* «Решил правильно»;
* «Нужно повторить».

После выбора:

* обновить localStorage;
* показать короткое уведомление;
* визуально обозначить выбранный статус;
* разрешить изменить решение позднее.

Если пользователь нажал «Решил правильно», случайная выдача должна исключать это задание по умолчанию.

Если пользователь нажал «Нужно повторить», задание появляется на странице `/review`.

# 12. Административная авторизация

## 12.1 Страница `/admin/login`

Поля:

* логин;
* пароль;
* кнопка «Войти».

Добавить:

* проверку пустых полей;
* понятные сообщения об ошибках;
* ограничение частоты неудачных попыток;
* блокировку кнопки во время отправки;
* безопасную cookie-сессию.

## 12.2 Данные администратора

Используй переменные окружения:

```env
ADMIN_LOGIN=admin
ADMIN_PASSWORD_HASH=...
SESSION_SECRET=...
DATABASE_URL=...
CRON_SECRET=...
UPLOAD_DIR=...
```

Не храни открытый пароль:

* в исходном коде;
* в Git;
* в клиентском JavaScript;
* в базе в незашифрованном виде.

Добавь скрипт для создания хеша пароля:

```bash
npm run hash-password
```

## 12.3 Сессия

Сессия должна:

* храниться в HTTP-only cookie;
* иметь флаг `secure` в production;
* иметь `sameSite=strict` или безопасный эквивалент;
* иметь ограниченный срок действия, например 8 часов;
* проверяться на всех административных страницах и API;
* удаляться при выходе.

Добавь кнопку «Выйти».

Проверка доступа должна происходить на сервере, а не только скрытием элементов интерфейса.

# 15. Формат CSV

CSV должен поддерживать UTF-8 и UTF-8 с BOM.

Поддержи разделители:

* запятая;
* точка с запятой.

Разделитель нужно определять автоматически.

## 15.1 Колонки

Обязательные:

* `category`;
* `difficulty`;
* `condition`;
* `expression_latex`;
* `answer`;
* `solution`.

Необязательные:

* `number`;
* `source`;
* `published`;
* `image_filenames`;
* `image_alt_texts`.

## 15.2 Пример

```csv
number;category;difficulty;condition;expression_latex;answer;solution;source;published;image_filenames;image_alt_texts
101;Неопределённые интегралы;easy;"Найдите неопределённый интеграл";"\int x^2\,dx";"$\frac{x^3}{3}+C$";"Используем формулу:\n\n$$\int x^n dx=\frac{x^{n+1}}{n+1}+C$$\n\nСледовательно:\n\n$$\int x^2dx=\frac{x^3}{3}+C$$";"Демонстрационная база";true;;
102;Двойные интегралы;medium;"Вычислите двойной интеграл";"\int_0^1\int_0^2(x+y)\,dy\,dx";"$3$";"Сначала вычислим внутренний интеграл:\n\n$$\int_0^2(x+y)\,dy=2x+2$$\n\nЗатем:\n\n$$\int_0^1(2x+2)\,dx=3$$";"Демонстрационная база";true;"area-102.png";"Область интегрирования"
```

## 15.3 Допустимые значения сложности

Принимать:

* `easy`, `лёгкий`, `легкий`;
* `medium`, `средний`;
* `hard`, `сложный`.

Перед сохранением преобразовывать к enum.

## 15.4 Опубликованность

Принимать:

* `true`, `1`, `yes`, `да`;
* `false`, `0`, `no`, `нет`.

Если поле не указано, импортировать как `HIDDEN`, чтобы администратор сначала проверил задание.

## 15.5 Категория

Разрешить указывать:

* точное название;
* slug.

Категория должна уже существовать.

Не создавай категории автоматически из CSV, чтобы опечатка не создала неправильную категорию.

## 15.6 Номер

Если номер указан:

* он должен быть положительным целым;
* он не должен повторяться в файле;
* он не должен существовать в базе.

Если номер не указан:

* назначить следующий свободный номер;
* номера назначать последовательно;
* учитывать номера, занятые другими строками этого же импорта.

## 15.7 Несколько изображений

Имена разделять символом `|`.

Пример:

```text
area.png|graph.png
```

Альтернативные описания задаются в том же порядке:

```text
Область интегрирования|График функции
```

Количество описаний может быть меньше количества изображений, но администратор должен получить предупреждение. Отсутствие alt-текста не должно отменять импорт, если остальные данные правильные.

# 16. Импорт CSV и изображений

## 16.1 Режимы импорта

Поддержи:

### CSV без изображений

Администратор загружает один CSV-файл.

### ZIP с изображениями

ZIP должен иметь структуру:

```text
integral-hub-import.zip
├── tasks.csv
└── images
    ├── area-102.png
    ├── graph-205.webp
    └── region-310.svg
```

Внутри ZIP должен находиться ровно один основной CSV-файл `tasks.csv`.

## 16.2 Страница `/admin/import`

Этапы:

1. Выбор CSV или ZIP.
2. Загрузка во временную директорию.
3. Проверка структуры.
4. Полная проверка строк.
5. Предварительный просмотр.
6. Подтверждение.
7. Импорт одной транзакцией.
8. Итоговый отчёт.

До подтверждения не создавать задания в основной базе.

## 16.3 Предварительный просмотр

Показывать:

* название файла;
* размер;
* количество строк;
* количество новых заданий;
* первые 20 строк;
* назначенные автоматические номера;
* найденные изображения;
* предупреждения;
* ошибки.

Если есть хотя бы одна критическая ошибка:

* блокировать кнопку импорта;
* не записывать ни одного задания;
* показать номер строки;
* показать название поля;
* показать понятное описание проблемы.

## 16.4 Полная отмена при ошибке

Импорт должен быть атомарным.

Если найдена ошибка хотя бы в одной строке:

* не импортировать ни одной строки;
* не переносить изображения в постоянное хранилище;
* удалить временные файлы;
* показать полный список найденных ошибок.

Если ошибка возникает во время записи:

* откатить транзакцию;
* удалить уже перемещённые файлы;
* не оставлять частично импортированные данные.

# 18. Проверка LaTeX и Markdown

Основная формула `expression_latex` должна проверяться через KaTeX до сохранения.

Для полей:

* `answer`;
* `solution`;
* `condition`;

нужно найти математические блоки:

* `$...$`;
* `$$...$$`;
* `\(...\)`;
* `\[...\]`;

и отдельно проверить каждую формулу.

При ошибке показывать:

* строку CSV;
* поле;
* проблемный фрагмент;
* сообщение KaTeX.

Markdown должен быть безопасным:

* не исполнять HTML;
* не исполнять JavaScript;
* не разрешать опасные URL;
* санитизировать результат.

# 19. Проверка дубликатов

Считать задание дубликатом на основании нормализованного сочетания:

* категория;
* условие;
* основная формула;
* ответ.

Перед сравнением:

* убрать лишние пробелы;
* привести переносы строк к одному виду;
* привести регистр обычного текста;
* нормализовать внешние пробелы;

В отчёте показывать:

* причину совпадения.

# 20. Хранение изображений

Для первой версии реализуй файловое хранилище через абстрактный сервис.

Интерфейс сервиса должен поддерживать:

* сохранение;
* чтение;
* удаление;
* перемещение из временного хранилища;
* проверку существования.

В режиме разработки используй локальную директорию:

```text
storage/uploads
```

Не сохраняй загруженные файлы непосредственно в Git.

Структура может быть:

```text
storage/uploads/tasks/{taskId}/{generatedName}.png
```

Предусмотри возможность позднее заменить локальное хранилище на S3-совместимое без переписывания бизнес-логики.

# 21. Очистка архива

При архивировании:

```text
archivedAt = текущее время
purgeAt = текущее время + 24 часа
```

Создай серверную функцию:

```text
purgeExpiredArchivedTasks()
```

Она должна:

1. найти архивные задания с `purgeAt <= now`;
2. удалить связанные изображения из хранилища;
3. удалить записи в базе;
4. логировать результат.

Вызывай очистку:

* при открытии административного архива;
* после административных операций;
* через защищённый cron-маршрут.

Создай маршрут наподобие:

```text
POST /api/cron/purge-archive
```

Он должен требовать секрет `CRON_SECRET`.

В README объясни, что для гарантированного удаления ровно после 24 часов в production необходимо настроить периодический вызов cron-маршрута. Без внешнего планировщика также должна работать отложенная очистка при следующем административном запросе.

# 29. Тестирование

* валидацию имён файлов;

## Интеграционные тесты

Проверить:

* успешное создание задания;
* запрет доступа без административной сессии;
* успешный импорт;
* полную отмену импорта при ошибке;
* отсутствие частично сохранённых заданий;
* восстановление из архива;
* окончательное удаление;
* удаление изображений вместе с заданием.

## Сквозные тесты

Проверить сценарии:

1. Пользователь открывает каталог.
2. Фильтрует двойные интегралы средней сложности.
3. Открывает задание.
4. Показывает ответ.
5. Показывает решение.
6. Отмечает «Нужно повторить».
7. Видит задание в разделе повторения.
8. Позднее отмечает «Решил правильно».
9. Задание исчезает из повторения.

Административный сценарий:

1. Администратор входит.
2. Загружает корректный CSV.
3. Просматривает предварительный результат.
4. Подтверждает импорт.
5. Видит новые задания.
6. Архивирует одно задание.
7. Восстанавливает его.

Ошибочный сценарий:

1. Администратор загружает файл со 100 строками.
